
# Next.js란 무엇인가  
  
## 한 줄 정의  
- Next.js는 React(React: 사용자 인터페이스 구축용 라이브러리) 기반의 풀스택(Full Stack: 프론트엔드와 백엔드를 함께 다루는 방식) 웹 프레임워크(Framework: 구조와 규칙을 제공하는 개발 틀)임  
  
## 왜 필요한가  
- React만으로는 라우팅, 서버 처리, 배포 구조를 직접 많이 구성해야 했음  
- Next.js는 페이지 구성, 서버 실행, 데이터 처리, 최적화를 한 프레임 안에서 다루기 좋게 만들었음  
  
## 계층 구조  
```text  
[사용자]  
↓  
[브라우저]  
↓  
[React 컴포넌트(Component: 화면을 구성하는 재사용 가능한 조각)]  
↓  
[Next.js]  
├─ 라우팅(Routing: 주소에 따라 화면을 연결하는 방식)  
├─ 렌더링(Rendering: 화면을 만들어 보여주는 과정)  
├─ 서버 기능(Server Function: 서버에서 실행되는 기능)  
├─ 데이터 처리(Data Fetching: 데이터 가져오기)  
└─ 최적화(Optimization: 더 빠르고 효율적으로 동작하게 만드는 작업)  
↓  
[데이터베이스(Database: 구조화된 데이터 저장소) / 외부 서비스]
```
## 핵심 개념
- React 위에서 동작함
- 서버 기능도 함께 다룰 수 있음
- 파일 기반 라우팅(File-based Routing: 파일 구조로 경로를 정하는 방식)을 사용함
- 렌더링 전략을 여러 방식으로 가져갈 수 있음
```text
React
→ 화면을 만들기 좋은 도구

Next.js
→ React에
   라우팅(Routing: 주소 연결),
   서버 처리(Server-side Processing: 서버 쪽 처리),
   렌더링(Rendering: 화면 생성),
   배포 구조(Deployment Structure: 실제 서비스 배치 구조),
   최적화(Optimization: 성능 개선)
   실서비스용으로 만든 프레임워크(Framework: 구조와 규칙을 제공하는 개발 틀)
```
 - React는 **화면을 잘 만드는 도구**였음
- 그런데 실제 서비스는 화면만으로 끝나지 않음
- 주소 체계, 데이터 불러오기, 서버 처리, 검색 엔진 대응, 이미지 최적화, 배포 구조까지 같이 필요했음
- Next.js는 이런 것들을 **처음부터 같이 제공하는 방향**으로 만들어진 도구임

## Next.js를 한 문장씩 분해해서 이해하기
### 1) React 기반임
- Next.js는 React 위에서 동작함
- 즉, React 문법과 컴포넌트(Component: 화면을 구성하는 재사용 가능한 조각) 개념을 그대로 사용함
- React를 실무형으로 확장한 느낌에 가까움
### 2) 페이지 구조를 쉽게 잡게 해줌
- 폴더와 파일을 기준으로 주소를 연결해줌
- 예를 들어 파일을 만들면 그 파일이 곧 페이지가 되는 방식으로 관리할 수 있음
- 그래서 라우팅(Routing: 주소에 따라 화면을 연결하는 방식)을 직접 복잡하게 짜는 부담이 줄어듦
### 3) 서버 기능도 같이 다룰 수 있음
- 브라우저에서만 동작하는 구조가 아니라 서버에서도 코드를 실행할 수 있음
- 그래서 데이터베이스(Database: 구조화된 데이터 저장소) 조회, 비밀 키 처리, 로그인 처리 같은 일을 할 수 있음
### 4) 화면을 만드는 방법이 다양함
- 요청이 들어올 때 서버가 바로 화면을 만들 수도 있음
- 미리 정적인 페이지를 만들어 둘 수도 있음
- 필요한 부분만 브라우저에서 다시 동작하게 만들 수도 있음
- 즉, 서비스 목적에 따라 전략을 바꿀 수 있음
### 5) 성능과 최적화 기능이 내장되어 있음
- 이미지 최적화
- 폰트 최적화
- 코드 분할(Code Splitting: 필요한 코드만 나눠서 불러오는 기법)
- 캐시(Cache: 다시 쓰기 위해 저장해 두는 데이터) 활용
- 이런 것들을 기본 구조 안에서 다룰 수 있음
## React와 차이
- React는 UI 라이브러리 중심 → 기본적으로 사용자 인터페이스(User Interface: 사용자 화면과 조작 요소) 중심
- Next.js는 라우팅, 서버 처리, 빌드, 최적화까지 포함한 실행 구조임

| 항목                                              | React                                            | Next.js                                  |
| ----------------------------------------------- | ------------------------------------------------ | ---------------------------------------- |
| 정체                                              | 라이브러리(Library: 필요한 기능을 </br>가져다 쓰는 도구 모음)        | 프레임워크(Framework: </br>구조와 규칙을 제공하는 개발 틀) |
| 중심                                              | 사용자 인터페이스(User Interface: </br>사용자 화면과 조작 요소) 구성 | 실서비스 전체 구조                               |
| 라우팅(Routing: 주소에 따라 </br>화면을 연결하는 방식)           | 직접 구성하는 경우 많음                                    | 기본 구조 제공                                 |
| 서버 기능                                           | 별도 구성 필요                                         | 같이 다루기 쉬움                                |
| 렌더링(Rendering: 화면을 </br>만들어 보여주는 과정) 전략         | 직접 선택과 구성 필요                                     | 기본 지원이 강함                                |
| 최적화(Optimization: 더 빠르고 </br>효율적으로 동작하게 만드는 작업) | 직접 챙길 부분 많음                                      | 내장 기능 많음                                 |
## 언제 쓰는가
- 검색 노출이 필요한 사이트
- 서버 데이터가 중요한 서비스
- 관리자 페이지 + 공개 페이지가 함께 있는 서비스
- 배포와 구조를 표준화하고 싶은 프로젝트
### 1) Next.js가 해결하려는 문제
```text
1. 주소가 늘어나면 페이지 관리가 복잡해짐  
   ↓  
   파일 기반 라우팅(File-based Routing: 파일 구조로 경로를 정하는 방식)으로 단순화  
2. 서버가 필요한 기능이 많아짐  
   ↓  
   서버 기능(Server Function: 서버에서 실행되는 기능)을 같은 프로젝트 안에서 처리 가능  
3. 검색 엔진 대응이 중요해짐  
   ↓  
   서버에서 미리 화면을 만들 수 있음  
4. 성능이 중요해짐  
   ↓  
   이미지, 폰트, 캐시(Cache: 다시 쓰기 위해 저장해 두는 데이터) 등을 체계적으로 관리 가능  
5. 배포 구조가 제각각이면 유지보수가 어려움  
   ↓  
   정해진 패턴으로 프로젝트를 운영 가능
```
### 2) Next.js를 쓰면 좋은 경우
```text
[콘텐츠 사이트]  
→ 블로그, 문서 사이트, 회사 소개 페이지  
[서비스 사이트]  
→ 로그인, 게시판, 대시보드, 관리자 페이지  
[검색 노출이 중요한 사이트]  
→ 서버에서 미리 화면을 보여주는 전략이 유리함  
[프론트엔드와 백엔드를 너무 분리하고 싶지 않은 경우]  
→ 한 프로젝트 안에서 같이 다루기 좋음
```
#### 예시
- 회사 홈페이지
- 기술 블로그
- 쇼핑몰 프론트엔드
- 관리자 시스템
- 로그인 기반 웹 서비스
- 게시판형 사이트
- 데이터 조회 중심 대시보드

## Next.js를 왜 실무에서 많이 쓰는가
### 이유 1) 구조가 통일되기 쉬움
- 팀 프로젝트를 하면 폴더 구조와 역할 분리가 중요함
- Next.js는 기본 흐름이 비교적 정해져 있어서 협업에 유리함
### 이유 2) 검색 엔진 대응에 유리함
- 브라우저에서만 늦게 만드는 화면보다, 서버에서 미리 준비한 화면이 유리한 경우가 많음
- 특히 콘텐츠 사이트나 소개 페이지에서 장점이 큼
### 이유 3) 프론트엔드와 서버 경계를 자연스럽게 다룸
- 완전히 따로 나누지 않아도 되는 프로젝트에서는 생산성이 좋음
- 작은 팀이나 빠르게 만들어야 하는 서비스에 잘 맞음
### 이유 4) 최적화 기능이 많음
- 이미지 처리, 글꼴 처리, 코드 분할(Code Splitting: 필요한 코드만 나눠서 불러오는 기법) 같은 부분을 기본 체계 안에서 다루기 좋음

## 자주 헷갈리는 점
- Next.js = React 대체제가 아님
- Next.js는 React를 기반으로 더 많은 기능을 묶어 둔 도구임
- Next.js는 React를 실서비스에 맞게 확장한 구조임
- 화면만 만드는 도구가 아니라 서버와 운영까지 이어지는 흐름을 같이 다룸
- 처음에는 파일 구조가 규칙처럼 느껴질 수 있는데, 익숙해지면 오히려 관리가 쉬워짐
### Next.js를 처음 배우는 사람이 꼭 알아야 하는 핵심 단어
#### 1) 라우팅(Routing: 주소에 따라 화면을 연결하는 방식)
- 사용자가 `/about` 같은 주소로 들어오면 어떤 페이지를 보여줄지 정하는 규칙임
#### 2) 렌더링(Rendering: 화면을 만들어 보여주는 과정)
- 화면을 **언제**, **어디서**, **어떻게** 만들지 정하는 개념임
#### 3) 서버 컴포넌트(Server Component: 서버에서 실행되는 컴포넌트)
- 서버에서 실행되어 화면을 준비하는 컴포넌트(Component: 화면을 구성하는 재사용 가능한 조각)임
#### 4) 클라이언트 컴포넌트(Client Component: 브라우저에서 실행되는 컴포넌트)
- 클릭, 입력, 상태 변화 같은 브라우저 상호작용을 처리하는 컴포넌트(Component: 화면을 구성하는 재사용 가능한 조각)임
#### 5) 데이터 가져오기(Data Fetching: 데이터 가져오기)
- 외부 서비스나 데이터베이스(Database: 구조화된 데이터 저장소)에서 필요한 데이터를 읽어오는 과정임
#### 6) 배포(Deploy: 실제 서버나 플랫폼에 올리는 과정)
- 만든 프로젝트를 실제 사용자들이 접속할 수 있게 올리는 작업임
