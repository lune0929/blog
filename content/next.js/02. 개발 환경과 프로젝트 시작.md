# 개발 환경과 프로젝트 시작  
  
## 한 줄 정의  
- Next.js 프로젝트를 만들고 실행하는 시작 단계임  
  
## 준비물  
### 1) Node.js(Node.js: 자바스크립트를 서버에서 실행하는 런타임)
- Next.js 프로젝트를 만들고 실행할 때 필요함
- 패키지 설치, 개발 서버 실행, 빌드 모두 Node.js 위에서 돌아감
### 2) 패키지 매니저(Package Manager: 라이브러리 설치와 실행을 관리하는 도구)
- npm(npm: Node Package Manager, 노드 패키지 관리자)
- pnpm(pnpm: Performant Node Package Manager, 성능 중심 노드 패키지 관리자)
- yarn(Yarn: 자바스크립트 패키지 관리자)
- bun(Bun: 자바스크립트 런타임 겸 패키지 관리자)
- 여기서는 가장 무난하게 npm 기준으로 설명함
### 3) 편집기
- VS Code(Visual Studio Code: 마이크로소프트 코드 편집기) 많이 씀
### 4) 브라우저
- 로컬 개발 서버 결과 확인용임
## 이 단계에서 꼭 알아야 하는 것
1. Next.js는 그냥 설치만 하면 끝나는 도구가 아님  
2. 프로젝트 폴더 구조가 곧 동작 규칙이 됨  
3. 특히 app 폴더, layout.tsx, page.tsx가 핵심 시작점임  
4. 그래서 처음 만들 때 구조를 같이 이해해야 나중에 안 꼬임

## 패키지 매니저
 - 패키지 매니저(Package Manager: 패키지 설치와 관리를 하는 도구)는 프로젝트에 필요한 라이브러리(Library: 필요한 기능을 가져다 쓰는 도구 모음)를 설치하고 버전을 관리하는 도구임
### 종류  
  
```text  
npm(npm: Node Package Manager, 노드 패키지 관리자)  
↓  
Node.js 설치 시 같이 들어오는 기본 도구  
  
pnpm(pnpm: Performant Node Package Manager, 성능 중심 패키지 관리자)  
↓  
설치 속도, 디스크 효율, 워크스페이스(Workspace: 여러 패키지를 함께 관리하는 작업공간) 관리에 강점이 있음  
  
yarn(Yarn: 자바스크립트 패키지 관리자)  
↓  
npm 대안으로 많이 쓰였던 도구임
```
### pnpm에서 자주 쓰는 명령  
  
```bash  
pnpm install                     ## 의존성 설치
pnpm approve-builds              ## 설치 스크립트 실행 승인 
pnpm dev                         ## 개발 서버 실행
pnpm audit                       ## 취약점 점검
pnpm audit --fix                 ## 가능한 범위의 취약점 대응 시도 package.json에 overrides 추가함
pnpm update                      ## 의존성 버전 업데이트 next 15.5.12 → 16.2.0 으로 올라감
```
### pnpm install
- `pnpm install`은 프로젝트에 필요한 모든 의존성을 설치하는 기본 명령임. 별칭(alias: 짧은 대체 이름)으로 `pnpm i`도 있음. 워크스페이스(workspace: 여러 패키지를 함께 관리하는 작업공간) 안에서는 기본적으로 모든 프로젝트의 의존성을 설치함.
- `pnpm install --frozen-lockfile` → 락파일을 바꾸지 않음
- `pnpm install --lockfile-only` → `node_modules`는 안 건드리고 락파일만 갱신함
- `pnpm install --offline` → 로컬 스토어(store: 패키지 저장 캐시)만 사용함
- `pnpm install --prod` → `devDependencies` 없이 운영용 의존성만 설치함.
- 로컬 개발은 그냥 `pnpm install`
- CI나 배포 환경은 `pnpm install --frozen-lockfile`을 많이 씀
- 모노레포(monorepo: 여러 패키지를 한 저장소에서 관리하는 구조)에서는 ~={red}루트에서 실행하면 전체 설치가 일어날 수 있음=~.
```text  
package.json 확인  
↓  
pnpm-lock.yaml 확인  
↓  
필요한 패키지 버전 결정  
↓  
node_modules 생성 또는 갱신  
↓  
일부 패키지의 build script(build script: 설치 중 실행되는 스크립트) 발견 
↓  
pnpm이 자동 실행하지 않고 일단 보류함   
↓  
승인이 필요한 경우 approve-builds 안내 출력

## pnpm approve-builds는 왜 뜨는가  
  - `pnpm approve-builds`는 설치 중 실행되는 스크립트를 허용할지 검토하는 보안 절차임 
  - 공급망 보안(supply chain security: 의존성 유통망 보안) 관점에서 한 번 걸러주는 흐름임. 
    공식 문서 기준으로 승인된 항목은 `pnpm-workspace.yaml`에 저장 

프로젝트 루트  
├─ package.json        → 어떤 패키지를 쓰는지 적는 파일임
├─ pnpm-lock.yaml      → 실제로 어떤 버전이 설치됐는지 고정하는 파일임
├─ node_modules/       → 설치된 패키지가 들어가는 폴더임
└─ app/
```

## 폴더와 파일 역할
#### `package.json`
- `package.json`은 프로젝트의 이름, 스크립트(script: 실행 명령 정의), 의존성(Dependency: 프로젝트가 동작하기 위해 필요한 외부 패키지), 각종 설정을 담는 **프로젝트 설명서** 같은 파일임. pnpm 공식 문서도 `package.json`을 패키지의 메타데이터(metadata: 설명 정보)를 담는 매니페스트(manifest: 선언 파일)라고 설명함.
- 프로젝트 정보와 의존성(Dependency: 프로젝트가 동작하기 위해 필요한 외부 패키지) 목록, 실행 스크립트가 들어 있음
```json
{
  "name": "my-next-app",          → 프로젝트 이름
  "version": "0.1.0",             → 프로젝트 버전
  "private": true,                → 배포용 패키지가 아닌지 여부
  "scripts": {                    → 실행 명령 모음
    "dev": "next dev",
    "build": "next build",
    "start": "next start",
    "lint": "eslint"
  },
  "dependencies": {               → 운영 시 필요한 패키지
    "next": "16.2.0",
    "react": "19.2.0",
    "react-dom": "19.2.0"
  },
  "devDependencies": {            → 개발 도구 패키지
    "typescript": "5.9.3"
  }
}

1. 프로젝트 생성  
2. npm run dev        → 개발 서버 실행  
3. app/page.tsx 수정 
4. 브라우저에서 확인
5. 폴더를 추가해서 새 페이지 생성
6. layout.tsx 에 공통 구조 추가
7. npm run build      → 배포용 결과물 생성  
8. npm run start      → 빌드된 결과를 운영 모드로 실행  
9. npm run lint       → 코드 규칙 검사
```

#### `pnpm-lock.yaml`
- `pnpm-lock.yaml`은 **실제로 설치하기로 확정된 의존성 버전 결과표**임. `pnpm install`은 락파일(lockfile: 설치 버전을 고정하는 파일)을 사용하고, CI(Continuous Integration: 지속적 통합) 환경에서는 락파일이 있는데 갱신이 필요하면 설치를 실패시킴.
- 같은 `package.json`이라도 설치 시점마다 세부 하위 의존성이 달라질 수 있음    
- 락파일이 있으면 팀원, 서버, CI에서 더 비슷한 결과를 재현하기 쉬움
- 락파일이 없으면 `pnpm install`이 새로 만듦
- 보통 Git에 같이 커밋(commit: 버전 관리 저장)함    
- 팀 프로젝트에서는 `package.json`만 올리고 락파일을 빼면 환경 차이가 커질 수 있음

#### `node_modules/`
- `node_modules`는 설치된 패키지가 보이는 폴더임. 다만 pnpm은 npm과 같은 평평한(flat: 납작한) 구조 대신, **하드 링크(hard link: 같은 실제 파일을 가리키는 파일 시스템 링크)와 심볼릭 링크(symbolic link: 다른 경로를 가리키는 링크)** 를 써서 구조를 만듦.
- pnpm은 `node_modules/.pnpm` 아래에 패키지를 구성하고, 루트 `node_modules`에는 직접 의존성 위주로 링크를 둠    
- 공식 문서도 pnpm의 `node_modules` 레이아웃이 심볼릭 링크 기반 구조라고 설명함.
- `node_modules`는 보통 ~={red}Git에 올리지 않음=~
- 깨졌다고 의심되면 지우고 `pnpm install`로 다시 만드는 경우가 많음

#### `next/`
- Next.js가 dev(build 전 개발 실행) 또는 build(배포용 빌드) 때 자동 생성하는 출력 폴더
```text
[소스 코드 폴더]                       직접 수정하는 원본은 `src`, `public`, 설정 파일들
src/app  
src/components  
src/lib  
public  
↓  
[Next.js가 해석하고 변환]  
↓  
[실행 가능한 결과물]                     실행용으로 가공된 결과
.next
```

#### `app/`
- 화면과 라우팅의 중심 폴더임
- App Router 기준에서 가장 중요함
- 폴더 구조가 URL 구조와 연결됨
#### `app/layout.tsx`
- 최상위 공통 틀임
- 루트 레이아웃(Root Layout: 최상위 공통 레이아웃)임
- `html`, `body` 태그를 반드시 포함해야 함
```typescript title:"가장 최소한의 시작 소스"
import './globals.css'
import Header from '@/components/Header'

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="ko">
      <body>
        <Header />
        <main>{children}</main>
      </body>
    </html>
  )
}

### 설명
- `children`은 하위 페이지가 들어오는 자리임
- 여기서는 `/`에 해당하는 `app/page.tsx`가 들어오게 됨
- 나중에는 헤더, 푸터, 공통 네비게이션(Navigation: 화면 이동 메뉴)도 여기에 넣게 됨
```
#### `app/page.tsx`
- 루트 경로 `/`에 해당하는 첫 페이지임
- 홈페이지 첫 화면이라고 보면 됨
```typescript title:"app/page.tsx"
export default function HomePage() {
  return (
    <section>
      <h2>홈 화면임</h2>
      <p>여기가 첫 진입 페이지임</p>
    </section>
  )
}
```

```typescript title:"components/Header.tsx"
export default function Header() {
  return (
    <header>
      <h1>My Next App</h1>
      <nav>
        <a href="/">홈</a>
        <a href="/about">소개</a>
      </nav>
    </header>
  )
}
```

```typescript title:"app/about/page.tsx"
export default function AboutPage() {
  return (
    <section>
      <h2>소개 페이지임</h2>
      <p>about 폴더 아래 page.tsx라서 /about 주소가 됨</p>
    </section>
  )
}
```
#### `app/globals.css`
- 전역 스타일 파일임
- 전체 페이지에 공통 적용되는 CSS(Cascading Style Sheets: 계단식 스타일 시트) 넣는 곳임
#### `public/`
- 이미지, 폰트, 아이콘 같은 정적 파일 넣는 곳임
- `/파일명` 형태로 접근함
- `public/profile.png`라고 저장했어도 코드에서는 `/profile.png`로 접근함
- `public`은 URL 기준 루트처럼 동작함
```typescript
import Image from 'next/image'

export default function ProfilePage() {
  return (
    <div>
      <Image
        src="/profile.png"
        alt="프로필 이미지"
        width={120}
        height={120}
      />
    </div>
  )
}
```

#### `next.config.ts` 또는 `next.config.js`
- Next.js 프로젝트의 전역 동작 방식을 정하는 설정 파일임
- **프로젝트 루트**에 두는 설정 파일이고, **Next.js 서버와 빌드 단계**에서 사용되며, **브라우저 번들(build: 배포 가능한 결과물로 묶는 과정)** 에 포함되지 않음.
```text
src/app/page.tsx               → 화면 내용 작성
src/components/*  
public/*  
↓  
이 소스들을 Next.js가 읽음  
↓  
next.config.ts 설정 참고         → 프로젝트 전체 동작 규칙 설정
↓  
dev/build/start 동작 방식 결정
```
1. output(Output: 빌드 결과물 형태) : standalone 출력
	- `output: 'standalone'`은 배포용으로 필요한 파일만 모아 `.next/standalone` 출력물을 만들 때 쓰는 설정임
2. redirects(Redirects: 경로 이동 규칙)  : /old → /new
	- `/old`로 들어오면 `/new`로 보내는 규칙임
	- `permanent: true`라서 영구 이동으로 처리함
3. headers(Headers: 응답 헤더 규칙)  : 보안 헤더 추가
	- 모든 경로에 보안용 HTTP(HyperText Transfer Protocol: 웹 통신 규약) 헤더를 붙이는 예제임
	- `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy` 같은 헤더를 추가함
4. typescript(TypeScript 설정)  : 빌드 시 타입 오류 무시 안 함
	- `ignoreBuildErrors: false`는 타입 오류가 나면 빌드를 실패시키겠다는 뜻임
	- 실무에서는 기본적으로 이쪽이 더 안전함
5. sassOptions(Sass Options: Sass 전처리기 설정)  : Sass 설정
	- Sass(Sass: CSS 전처리기) 공통 변수나 설정을 주입할 때 쓰는 예제임
6. cacheComponents(Cache Components: 컴포넌트 캐시 기능) : 컴포넌트 캐시 기능 켬
```ts  
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  output: 'standalone',

  async redirects() {
    return [
      {
        source: '/old',
        destination: '/new',
        permanent: true,
      },
    ]
  },

  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'Referrer-Policy',
            value: 'strict-origin-when-cross-origin',
          },
        ],
      },
    ]
  },

  typescript: {
    ignoreBuildErrors: false,
  },

  sassOptions: {
    additionalData: `$brand-color: #2563eb;`,
  },

  cacheComponents: true,
}

export default nextConfig
```

#### `tsconfig.json`
- `tsconfig.json`은 **TypeScript(TypeScript: 타입을 추가한 자바스크립트)** 가 프로젝트 코드를 어떻게 해석하고 검사할지 정하는 설정 파일임. Next.js는 TypeScript를 기본 지원하고, `create-next-app`으로 시작하면 필요한 설정을 자동으로 잡아주며, 기존 프로젝트도 `next dev`나 `next build`를 실행하면 필요한 의존성과 권장 `tsconfig.json`을 추가할 수 있음.
- 어떤 파일을 검사할지 정함
- 어떤 자바스크립트 문법 수준을 기준으로 볼지 정함
- 경로 별칭인 `@/*` 같은 걸 어떻게 해석할지 정함
- `next-env.d.ts` 같은 Next.js 타입 파일을 프로젝트에 포함시킴
- 엄격 모드(strict: 더 빡빡한 타입 검사) 여부를 정함
```json
{
  "compilerOptions": {
    "target": "ES2017",
    "lib": ["dom", "dom.iterable", "esnext"],
    "allowJs": true,
    "skipLibCheck": true,
    "strict": true,                              → 타입을 더 엄격하게 검사함
    "noEmit": true,                              → TypeScript가 별도 출력 파일을 직접 만들지 않음
    "esModuleInterop": true,
    "module": "esnext",
    "moduleResolution": "bundler",
    "resolveJsonModule": true,
    "isolatedModules": true,
    "jsx": "preserve",
    "incremental": true,
    "plugins": [
      {
        "name": "next"
      }
    ],
    "paths": {                       → @/components/Button 같은 경로를 쓸 수 있게 함
      "@/*": ["./src/*"]             → import Header from '@/components/Header' 사용 가능
    }
  },                                             → 어떤 파일을 타입 검사 대상에 넣을지 정함
  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx", ".next/types/**/*.ts"],
  "exclude": ["node_modules"]
}
```
#### `eslint.config.mjs`
- `eslint.config.mjs`는 **ESLint(ESLint: 자바스크립트와 타입스크립트 코드 규칙 검사 도구)** 의 검사 규칙을 정하는 설정 파일임. ESLint는 현재 flat config(flat config: 평평한 구조의 새 설정 방식)를 기본으로 쓰며, Next.js도 `eslint-config-next`를 통해 Next.js용 권장 규칙을 제공함.
- 코드 스타일과 규칙을 검사함
- 사용하지 않는 변수, 위험한 패턴, 잘못된 훅 사용 등을 잡음
- Next.js 전용 규칙을 적용할 수 있음
- 어떤 파일에 어떤 규칙을 적용할지 범위를 나눌 수 있음
- mjs → ECMAScript Module(ECMAScript Module: 표준 자바스크립트 모듈 방식) 형식 파일
```js
import { defineConfig, globalIgnores } from 'eslint/config'
import nextVitals from 'eslint-config-next/core-web-vitals'
import nextTs from 'eslint-config-next/typescript'

export default defineConfig([
  ...nextVitals,                      → Next.js + Core Web Vitals(웹 성능 핵심 지표) 권장 규칙
  {
    rules: {                          → 규칙 추가
      'no-console': 'warn',      
    },
  },  
  ...nextTs,                          → TypeScript 관련 Next.js 권장 규칙
  globalIgnores([                     → 검사하지 않을 폴더/파일 지정
    '.next/**',
    'out/**',
    'build/**',
    'next-env.d.ts',
  ]),
])
```

## 자주 헷갈리는 점
- 개발 서버와 배포 서버는 다름
- 의존성(Dependency: 프로젝트가 동작하기 위해 필요한 외부 패키지) 설치와 실행은 별개임
#### 1) React 프로젝트와 뭐가 다른가
- 일반 React만 쓸 때는 라우팅과 폴더 구조를 직접 더 많이 잡게 됨
- Next.js는 시작부터 규칙이 정해져 있어서 그 규칙에 맞춰 움직이는 편임
#### 2) `layout.tsx`와 `page.tsx` 차이가 뭔가
- `layout.tsx`는 공통 틀임
- `page.tsx`는 실제 화면 내용임
#### 3) `public`에 넣는 것과 import 하는 것은 뭐가 다른가
- `public`은 정적 파일 직접 제공용임
- 컴포넌트 내부 import는 번들(Bundle: 브라우저로 전달되는 묶음 코드) 과정과 연결되는 경우가 많음
- 처음에는 이미지처럼 외부 노출용 자원은 `public`에 둔다고 이해하면 편함
#### 4) `src` 폴더는 꼭 필요한가
- 아님
- 선택 사항임
- 다만 파일이 많아지면 `src` 아래로 코드 몰아 넣는 방식이 정리하기 편할 수 있음
#### 5) 처음부터 폴더를 세세하게 나눠야 하나
- 그럴 필요 없음
- 처음에는 `app`, `components`, `public` 정도만 써도 충분함
